# astro-bot Roadmap

產品路線圖的唯一來源。`README.md` 提供使用者面的功能摘要，`AGENTS.md` 提供開發 agent 的摘要，但路線圖優先順序異動必須在此處先反映。

---

## 產品願景

成為台灣天文攝影者的**智慧出勤顧問**。

不只回答天體位置，而是幫攝影者做出最省力、最划算的出勤決策：從「值不值得去」，到「去的話怎麼安排」。

---

## 產品原則

astro-bot 不是銀河計算機，也不是旅遊 App。它是**攝影出勤決策引擎**，核心問題永遠是：

> 「這次出勤值得嗎？如果值得，怎麼做最省力？」

支援範圍：

- 深空與星野：銀河、星座、星雲、星系、流星雨、彗星等
- 日月行星運行景象：月出月落、月相、日月方位、行星可見性、行星合月等
- 天氣與地景條件：雲量、透明度、視寧度、結露、霧、雲海、海岸潮汐等
- 攝影決策：去哪裡、何時拍、風險是什麼、備案地點與器材提醒
- 出勤規劃：幾點出發、住宿區域建議、多日最佳夜次選擇

---

## 不可妥協的原則（Operating Guardrails）

- **不猜測**：天文計算與座標必須來自程式碼或可信資料來源；LLM 只能解釋與摘要，不可發明天體位置、氣象、視寧度、潮汐或地點座標
- **氣象優先**：天候是第一道 Go/No-Go 關卡；天況差時必須給出清楚的 No-Go，不得在天文分析後輕描淡寫地補一句「但天氣不好」
- **攔截未知**：未支援或尚未整合的查詢必須攔截，明確說明限制，不讓 LLM 以推測填補
- **地點審核制**：只有 `review_status: "approved"` 的地點可進 production；未知地點請使用者補座標
- **後勤資訊免責**：住宿、道路、入場限制等後勤資訊應標示「建議自行確認最新狀態」，不以 Bot 作為即時保證來源
- **部署門禁**：所有 production 變更必須通過 `HERMAS_AGENT.md` 的 dry-run gate 後才可 commit/push/deploy

## KPI 框架

下列指標用於追蹤服務健康度與各階段成熟度。MAU 與查詢滿意率屬於 developer review index，用來判斷產品是否被使用、回覆是否值得信任；它們不作為 Phase 3A → 3B 的硬性 gate。

| 指標 | 定義 | 追蹤方式 |
| --- | --- | --- |
| MAU | 月活躍用戶數（當月至少發出一次查詢） | LINE User ID，Google Sheets 查詢記錄 |
| 查詢滿意率 | 👍 ÷ (👍 + 👎) | Google Sheets 用戶反饋 |
| P90 回應時間 | 90% 查詢從送出到收到完整回覆的秒數 | Render Logs |
| 30 日留存率 | 首次查詢後 30 天內再次查詢的用戶比率 | Google Sheets 查詢記錄 |
| 平均查詢頻率 | 每位 MAU 當月平均查詢次數 | Google Sheets 查詢記錄 |

## 版本現況（v2，2026-06）

已完成的功能：

- LINE Bot + Render Web Service，Flask Webhook 24 小時持續運行
- 自然語言中文查詢，不需要特定格式
- 完整 Messier 目錄 M1–M110（約 110 個天體，含 RA/Dec、設備標籤、季節分類、aliases）
- Skyfield 天文計算：仰角、方位角、月出月落、天文薄暮、有效暗空窗口、銀河構圖方位角
- Open-Meteo 氣象預報（15 天）：雲量、溫度、濕度、露點、風速、能見度
- CCI 出勤信心指數（0–100 分，六因子加權含風速）
- 紅藍軍對抗邏輯：反樂觀守則 + 風險旗幟強制輸出
- 地點比較模式（合歡山 vs 阿里山）：CCI、暗空窗口、雲量、結露並排比較
- 今晚 / 週末最佳地點排名：全台 / 區域 Top 6，支援銀河、流星雨等題材
- 多題材 CCI 框架：meteor / moonscape / lunar_eclipse / comet_layer1 四種 profile
- 設備適配標籤（min_focal_mm、tracking_required、difficulty）+ 回覆自動注入【裝備提醒】
- 未命中目標固定格式回覆 + 自動寫入 Google Sheets 反饋
- 結露風險提醒（T - Td < 1.5°C）
- 曝光建議（500 法則快門、依月相 ISO、光圈）
- 用戶評分 👍👎 與許願池 💡（Google Sheets）
- 多用戶隔離
- 自動重啟
- 113 個台灣 approved 地點（含 aliases）
- 15 天景點氣象日曆（服務選單功能）
- 快捷服務選單（`選單` / `/menu`）

## Phase 3：決策品質升級

**主題：從「能查到資料」→ 「能給出可信賴的出勤決策」**

**進入條件：** v2 已上線（已滿足）

---

### 3A：核心決策引擎（最高優先）

3A 全部完成後才開始 3B。

3A → 3B 的判斷以功能完成度與測試結果為主；MAU 與查詢滿意率只作為服務健康 review index，不作為 3A exit hard condition。

| 開發順序 | 功能 | 說明 |
| --- | --- | --- |
| 1 | 信心度指標（CCI） | 在 ✅/⚠️/❌ 之外加入「信心度 XX%」，量化預報不確定性，讓用戶知道「這個建議可以賭多少」 |
| 2 | 紅藍軍對抗邏輯 | 每次回覆必須明確列出「最強不去理由」；防止 LLM 過度樂觀，建立使用者信任 |
| 3 | 地點比較模式 | 支援「合歡山 vs 阿里山」查詢；比較項目：CCI、暗空窗口、月光干擾、雲量、結露、透明度 |
| 4 | 今晚 / 週末最佳地點 | 從 approved 地點中排名，回答「今晚哪裡最好拍」；注意同時計算多地點的效能影響 |
| 5 | 多題材 CCI 框架 | CCI 套用於流星雨、月景、月蝕；彗星提供第一層（氣象 CCI），不給方位角；日月行星、雲海、日落日出排除在本階段外（見 `SUBJECT_SCOPE.md`）。同步在天體資料庫加入設備適配標籤（最低焦段、是否需赤道儀、難度等級），深空目標查詢回覆時附上設備建議提示；用戶個人設備記憶與個人化過濾留至 Phase 4B |

---

### 3B：規劃 UX 與資料營運（3A 完成後進行）

| 開發順序 | 功能 | 說明 |
| --- | --- | --- |
| 1 | User State 持久化儲存 | 技術債：2026-06-10 架構檢討已列為最高優先（原建議應在 Phase 3 前完成），至今未處理。`main.py` 的 `user_state`、`user_pending_location_query`、`user_last_query`、`user_wish_text` 皆為記憶體 dict，Render 重啟或 redeploy 會直接清空使用中的對話狀態（例如正在等待補座標、正在跑 15 天氣象選單流程），且不會有任何錯誤提示。建議改為輕量持久化（檔案或 DB），越早處理越好，避免部署頻率提高後放大此問題 |
| 2 | 回覆速度優化 | 目標 P90 < 15 秒（現況 30–60 秒）；已完成：氣象/視寧度平行查詢、`[耗時]` 分段記錄、OpenRouter 模型改用 Gemini 2.5 Flash（`generate_reply` 降 77%）。**尚未完成**：開放探索型查詢（無指定標的）觸發 117 個標的暴力掃描，`compute_target_windows` 20+ 秒，已診斷根因（觀測者天文位置被逐標的重複計算，可共用）但暫緩實作，待評估後再處理 |
| 3 | ✅ 語音輸入支援 | **2026-07-14 完成並上線**。接收 LINE 音訊訊息（M4A/AAC-LC）→ 透過既有 `OPENROUTER_API_KEY` 呼叫 OpenRouter 多模態音訊輸入（`google/gemini-2.5-flash`，不需另申請 Gemini API key）→ 轉成文字後接續既有查詢流程；辨識信心不足（LLM 自評 low）時回覆「沒有聽清楚，請用文字輸入」，不將低信心結果送入查詢流程；回覆維持既有文字格式，不做語音回覆。已用真實 LINE 語音測試驗證：已知地點 happy path（confidence=high，語音處理僅 2.31s）與未知地點安全機制（正確觸發補座標流程，未因語音輸入繞過「不猜測」原則）皆通過。2026-07-14 調整為優先於原 #4–#11，使用者明確決定 |
| 4 | 地點資料庫擴充（後勤欄位） | 在 `taiwan_locations.json` 加入：Bortle 指數（光害等級）、海拔、入場限制、車程基準、停車資訊、附近補給城鎮、是否可露營 |
| 5 | 備案地點建議 | 低地起霧或本地雲量高時，推薦附近破霧高度以上的替代地點 |
| 6 | 多日最佳夜選擇 | 使用者輸入「這個週末」時，明確比較各夜並推薦最佳出勤夜次 |
| 7 | 器材建議 | 依地點（海拔、氣溫）與目標給出器材提醒（建議焦段、電池備量、防露帶、赤道儀需求） |
| 8 | 地點許願池審核流程 | 建立半自動流程：Google Sheets 許願 → 座標格式驗證 → 人工補資訊 → approved；高頻未命中地點優先處理 |
| 9 | 雲海 / 霧判斷模式 | 針對「想拍雲海」「會不會起霧」建立獨立判斷邏輯，使用濕度、露點差、海拔與地點特性 |
| 10 | 日月行星景象規劃 | 接入可靠天文資料來源與計算邏輯後才開放；未完成前維持攔截與「不猜測」原則 |
| 11 | 15 天日曆摘要優化 | 每日 Go/No-Go 排名 + Top 3 詳情，比現有純氣象表格更具決策意義 |
| 12 | 彗星即時座標整合（第二層） | 維護可觀測彗星清單 + 接入 MPC（Minor Planet Center）自動抓取當前亮彗星；每日凌晨 2 點更新當日 + 未來 15 天座標並快取（本地或 Google Sheets）；查詢時直接讀快取，快取失效時降級為第一層（氣象 CCI）並告知用戶；完成後開放彗星方位角與目標可見性 CCI 因子 |
| 13 | 視寧度資料來源升級（Meteoblue） | 評估 Meteoblue astronomy seeing API 對台灣山區的覆蓋精度與成本；確認優於 7Timer 後替換；Windy 不採用（無直接視寧度 API） |
| 14 | 流星雨 ZHR 即時修正（IMO Live） | 接入 IMO Live ZHR，於流星雨極大期提供即時 ZHR 修正，取代靜態歷史平均值；GMN 即時通量資料格式複雜，暫不考慮 |
| 15 | 南十字座地點限制強化 | `taiwan_locations.json` 加入 `southern_horizon_clear` 欄位；南十字座查詢同時檢查緯度 < 24.5°N 且 `southern_horizon_clear: true`；不符合條件地點回覆明確提示 |

## Phase 4：出勤規劃層（How to Go）

**主題：從「值不值得去」→「去的話怎麼安排最省力」**

**進入條件：** Phase 3 離開指標全部達到

---

### 4A：出勤規劃基礎（最高優先）

| 開發順序 | 功能 | 說明 |
| --- | --- | --- |
| 1 | 出發時間建議 | 根據用戶所在城市（或輸入出發地）+ 地點車程基準 + 暗空窗口開始時間，推算「建議幾點出發、幾點可離開返家」 |
| 2 | 靜態住宿策展 | 各景點附近建議住宿城鎮與類型（民宿聚落、露營地、車宿）；標示「請自行確認即時房況」；不接即時訂房 API，先驗證需求 |
| 3 | 道路與入場限制提醒 | 靜態資料：各地點特殊限制（嘉明湖入山許可、合歡山冬季封山條件）；明確標示「建議出發前確認最新公告」 |
| 4 | 完整出勤簡報 | 整合天況（CCI）+ 出發時間 + 住宿建議 + 器材提醒 + 入場注意，一次回覆完整出勤計劃 |

---

### 4B：個人化（4A 完成後進行）

| 開發順序 | 功能 | 說明 |
| --- | --- | --- |
| 1 | 用戶基本資料記憶 | 儲存常用出發城市、慣用地點；讓 Bot 不需每次詢問出發地 |
| 2 | 器材檔案記憶 | 儲存鏡頭焦段、相機型號、是否有赤道儀；根據用戶設備主動過濾不適合的深空目標，並個人化曝光計算與器材提醒（Phase 3A #5 已加入靜態設備適配標籤作為前置基礎） |
| 3 | 觀測計劃保存 | 使用者可儲存「6/20 合歡山銀河」，日後查詢時 Bot 自動更新條件並比較 |

## 待驗證後才做

在以下條件達到前不投入開發資源。

| 功能 | 驗證門檻 |
| --- | --- |
| 雲層分層資料來源（Clear Outside 等） | 確認台灣覆蓋範圍與穩定性，且許願池有 > 10 則相關需求 |

| 即時路況 / Google Maps 整合 | Phase 4 出發時間功能使用率確認後評估 |
| 潮汐數據 | 海岸類查詢占月總查詢 > 15% 時才做 |

## 永久不做

| 功能 | 原因 |
| --- | --- |
| 無人機飛安警告 | 拉入飛安法規責任，超出天文攝影決策核心定位 |
| 管制區資料庫 | 法規變動風險高；以一般提醒取代，不維護高責任資料庫 |
